# セキュリティ監査レポート（デプロイ版）

Jimoto-Connect の本番デプロイ状態におけるセキュリティ懸念点の洗い出しです。

**監査日**: 2025年2月21日

---

## 1. 重大度別サマリー

| 重大度 | 件数 | 概要 |
|--------|------|------|
| 高 | 2 | 投稿先の検証漏れ ✅、画像アップロードの検証 ✅ いずれも対応済み |
| 中 | 4 | レート制限なし、コンテンツ長制限なし、RLSとClerkの関係、環境変数 |
| 低 | 3 | ストレージ公開、エラーメッセージ、Webhook |

## 2. 高リスク

### 2.1 投稿先（schoolId）の検証漏れ ✅ 対応済み（2025年2月21日）

**場所**: `src/app/actions/post.ts`

**内容**: ユーザーが `schoolId` をフォームで送信するが、**そのユーザーがその学校に所属しているか**を検証していない。

**リスク**: 任意の学校の掲示板に、他校のユーザーが投稿できてしまう（なりすまし投稿・スパム）。

**対応**: `profile_schools` テーブルで `profile_id` と `school_id` の組み合わせを確認。所属していない場合は「投稿先の学校に所属していません」でエラー。

---

### 2.2 画像アップロードの検証不足 ✅ 対応済み（2025年2月21日）

**場所**: `src/app/actions/post.ts`

**内容**:
- **ファイル形式**: `fileExt = file.name.split('.').pop()` のみで判定。拡張子の偽装が可能（例: `malicious.exe` → `malicious.exe.jpg`）
- **ファイルサイズ**: 制限なし。大量アップロードでストレージ枯渇の可能性
- **MIMEタイプ**: 検証なし

**リスク**: 悪意あるファイルのアップロード、ストレージの不正使用。

**対応**: 許可拡張子（jpg, jpeg, png, gif, webp）・MIMEタイプ・5MB制限を実装済み

---

## 3. 中リスク

### 3.1 レート制限なし

**場所**: 全 Server Actions（post, comment, like, event, message など）

**内容**: 同一ユーザーからの連続リクエストに制限がない。

**リスク**: スパム、DDoS 的な負荷、API の濫用。

**対応案**:
- Vercel の Edge Middleware や Upstash 等でレート制限を実装
- または Supabase の Function で制限

---

### 3.2 コンテンツ長の制限なし

**場所**: post, comment, message, プロフィール更新など

**内容**: 投稿文・コメント・メッセージの文字数制限がない。

**リスク**: 極端に長いコンテンツによる DB 負荷、ストレージ圧迫。

**対応案**:
- 投稿: 例 5,000 文字
- コメント: 例 1,000 文字
- メッセージ: 例 2,000 文字

---

### 3.3 Supabase RLS と Clerk の関係

**場所**: マイグレーション（RLS ポリシー）、Supabase クライアント

**内容**:
- RLS ポリシーは `auth.jwt() ->> 'sub'` を前提（Supabase Auth 想定）
- 本アプリは **Clerk** 認証で、Supabase Auth は未使用
- 書き込みはほぼすべて **Service Role** を使用しており、RLS をバイパス
- セキュリティは **Clerk の auth() チェック** と **アプリ側のロジック** に依存

**リスク**: Service Role キーが漏洩した場合、RLS が効かないため DB 全体にアクセス可能。ただしキーはサーバー側のみで、クライアントには出ていない。

**対応案**:
- Service Role キーを Vercel の環境変数で厳重に管理（実施済み想定）
- 漏洩時は即座に Supabase でキーをローテーション

---

### 3.4 環境変数の管理

**内容**:
- `SUPABASE_SERVICE_ROLE_KEY` はクライアントに露出していない（`NEXT_PUBLIC_` なし）✅
- `.env*` は `.gitignore` に含まれている ✅
- Vercel の環境変数が本番用に正しく設定されているか要確認

**対応**: デプロイ前に Vercel の Environment Variables で本番用キーが設定されていることを確認。

---

## 4. 低リスク

### 4.1 ストレージ（post-images）の公開設定

**場所**: `supabase/migrations/20260216150000_create_storage_bucket.sql`

**内容**: `post-images` バケットは `public: true`。アップロードされた画像は URL を知っていれば誰でも閲覧可能。

**リスク**: 想定どおりの挙動だが、URL 推測による列挙の可能性はある（UUID ベースなら推測は困難）。

---

### 4.2 エラーメッセージの詳細表示

**内容**: 一部で `console.error` に詳細を出力。本番では Vercel のログに残る。

**リスク**: ログに DB エラーやスタックトレースが含まれる可能性。ログアクセス権限の管理が重要。

---

### 4.3 Clerk Webhook の未実装

**場所**: `src/middleware.ts` で `/api/webhooks(.*)` が公開ルートとして指定

**内容**: `src/app/api/` 配下に Webhook 用 API ルートが存在しない。

**リスク**: ユーザー削除時のプロフィール同期など、Clerk 連携の高度な機能が未実装。現状の認証フローでは必須ではない。

---
